1 Introduction

The stm32ai application is a console utility which provides a complete and unified Command Line Interface to perform the main steps to analyze, to validate and to generate an optimized NN C-library for STM32 device family. Each command can be used independently of the other with the same set of common options (model files, compression factor, output directory…) and also with the specific arguments.

This utility is used as back-end by the X-CUBE-AI UI plug-in (see [2])

1.1 Synopsis

usage: stm32ai    [-h] [--model FILE] [--version] [--verbosity [{0,1,2}]]
                  [--type [keras|tflite|caffe|convnetjs|lasagne]]
                  [--name [STR]] [--compression [1|4|8]] [--quantize [FILE]]
                  [--workspace [DIR]] [--output [DIR]] [--batches INT]
                  [--mode MODE] [--desc DESC] [--valinput FILE]
                  [--valoutput FILE] [--binary]
                  analyze|generate|validate|quantize

1.2 Comparison with the X-CUBE-AI UI plug-in features

In comparison with the X-CUBE-AI UI plug-in, the following features (see [2], section “STM32CubeMX extension”) are not supported through the stm32ai utility:

  • multiple DL models support is only available through the UI. The CLI allows only to analyze, to generate and to validate the specialized code for a simple DL model.
  • generation of the full IDE project including the expected network_runtime.a is only possible through the UI. The CLI can be only used to generate the specialized C-code files, however they can be used to update an initial IDE project (X-CUBE-MX based or proprietary configuration).
  • the check to indicate that a generated DL model fits in the device memory in term of memory layout is only performed by the UI. Only the system level integration metrics are reported: ROM, RAM, MACC..
  • for the validation process on target, as a full STM32 project is expected, it must be generated previously through the UI. Note that this project can be also updated when the initial DL model is modified through the CLI. Validation process on desktop is fully supported through the CLI without restriction.

1.3 Command work-flow

For each command, the following stages are applied. A report (txt file) is systematically created and displayed. There is also a well-formatted JSON file (dictionary based) in the workspace directory which can be used by a non-regression environment. The format of this file is out of the scope of this document.

<workspace-directory-path>\<name>_report.json
<output-directory-path>\<name>_<cmd_name>_report.txt
  • analyze flow
    • import the model
    • map, render and optimize internally the model
    • log and display a report
  • validate flow
    • import the model
    • map, render and optimize internally the model
    • execute the generated C-model (on desktop or on target)
    • execute the original model (if not quantized type)
    • evaluate the validation metrics, L2r error based or/and confusion matrix based
    • log and display a report
  • generate flow
    • import the model
    • map, render and optimize internally the model
    • export the specialized C-files
    • log and display a report

1.4 Error handling

During the execution of a given command, after the parsing of the arguments if an error is raised, the stm32ai application returns -1 else 0. The description of the error is displayed at the beginning of the summary.

An error message is prefixed by a category and a short description.

Category Description
LOAD ERROR error during the load/import of the model or the connection with the STM32 board
INVALID MODEL provided network model file is corrupted or cannot be parsed
INVALID OPTIONS specific model parameter is invalid
NOT IMPLEMENTED expected feature is not implemented
TOOLS ERROR INTERNAL ERROR internal error
CLI ERROR specific CLI error
INTERRUPT indicates that the execution of the command has been interrupted by the user (CTRL-C or kill system signal)

Example of error

...
Exec/report summary (analyze 0.000s err=-1)
------------------------------------------------------------------------------------------------------
error           : NOT IMPLEMENTED: Quantizing a compressed tensor is not supported for dense_4_weights
model file      : C:\ai_lab\fxp_demo\demo_kws_fxp\ds_cnn_Q.h5
type            : keras (keras_dump)
...

1.5 Setting the environment

X-CUBE-AI Expansion Package is a complete self-contained application package. No external tools is requested to use the package. %CUBE_FW_DIR% designates the root path where the X-CUBE expansion packages are installed (default location on Windows: C:\Users\<user_name>\STM32Cube\Repository)

1.5.1 Windows® 10 and Windows® 7

  1. Open a command prompt
  2. Update environment variables
set CUBE_FW_DIR=C:\Users\<user_name>\STM32Cube\Repository

set X_CUBE_AI_DIR=%CUBE_FW_DIR%\Packs\STMicroelectronics\X-CUBE-AI\4.0.0
set PATH=%X_CUBE_AI_DIR%\Utilities\windows;%PATH%
  1. Verify the environment
>  stm32ai --version
stm32ai - Neural Network Tools for STM32 v1.0.0 (AI tools v4.0.0)

or

>  stm32ai --tools_version

2 Common arguments

Following table describes the common options for the analyze, validate and generate commands. The specific arguments are described in the associated command section. The quantize command is a specific case (see “Quantization and quantize command” section).

Parameter Description
-m/--model indicates the original model file paths (see “DL framework detection” section) - Mandatory
-t/--type indicates the type of original DL framework when it can be not inferred by the extensions of the model files (see “DL framework detection” section) - Optional
-w/--workspace indicates a working/temporary directory for the intermediate files (default:"./stm32ai_ws/" directory) - Optional
-o/--output indicates the output directory for the generated and log files (default:"./stm32ai_output/"directory) - Optional
-n/--name indicates the C-name (C-string type) of the generated C-files (see [2]). It is also used as prefix for the generated files (including the temporary files). This allows to use the same workspace/output directories for different models (default: "network" name). - Optional
-c/--compression indicates the expected global factor of compression which will be applied. Supported values: 1|4|8 (default: ‘1’). See [2], section “Graph flow and memory layout optimizer”. Compression can be only performed on the dense-type layer. - Optional
-q/--quantize indicates file path of the tensor format configuration file (json file format) for the quantization process (see “Quantization and quantize command” section) - Optional

Note — The compression option is not supported for a quantized layer. If a global compression request is expected, the following error is generated: NOT IMPLEMENTED: Quantizing a compressed tensor is not supported for <name_layer>

3 analyze command

This command allows to import, to check and to render a DL model. Generated report provides the main expected system metrics to know if the generated code can be executed on a STM32 device. After this command, the user can be fully confident on the imported model in term of supported layer/operators.

For the floating-point models, the minimum and mandatory argument is to pass the model files if there is multiple files (topology and weights) --model/-m should be used for each file. For the quantized models, --quantize/-q argument should be also passed.

3.1 Command examples

  • Analyze a floating-point model (simple model file)

        $ stm32ai analyze -m <model_file_path>

  • Analyze a floating-point model (multiple model files, caffe type example)

        $ stm32ai analyze -m ./caffe/lenet.prototxt -m ./caffe/lenet.caffemodel

  • Analyze a floating-point model with compression request

        $ stm32ai analyze -m <model_file_path> -c 8

  • Analyze a quantized model (see “Quantization and quantize command” section)

        $ stm32ai analyze -m <modified_model_file>.h5 -q <quant_file_desc>.json

3.2 DL framework detection

Extension of the model files are used to identify the DL framework which should be used to import the model. If the detection is ambiguous, the --type/-t option should be used to define the correct framework.

DL framework type (--type/-t) file extension
Keras keras .h5 or .hdf5 and .json or .yml or yaml
TensorFlow lite tflite .tflite
Lasagne lasagne .npz and .py
Caffe caffe .prototxt and .caffemodel
ConvNetJS convnetjs .json

If you try to force a type which is not valid, typical following error is reported

$ stm32ai /c/ai_lab/fxp_demo/demo_asc_fxp/Session_keras_mod_93_Model.h5 --type tflite
...
TOOL ERROR: Invalid extension on input file
    C:\ai_lab\fxp_demo\demo_asc_fxp\Session_keras_mod_93_Model.h5

The second sub-section shows the results of the importing and rendering stages.

field description
model_name designates the logical name of the provided model. This is generally the name of the model file.
model_hash provides a calculated MD5 signature for the used model files.
input indicates the name, the item number, the format and the size in bytes of an input tensor. There is one line by input. input (total) field indicates the total size (in bytes) of the inputs.
o input_0 (item#=490, size=1.91 KiB, fmt=FLOAT32) indicates that input_0 tensor has a size of 490 floating-point items (size in bytes = 490 x 4B = 1.91KiB)
output indicates the name, the format and the size of the output tensor. There is one line by output. output (total) field indicates the total size (in bytes) of the outputs.
param # indicates the total number of parameters from the original model
macc indicates the logical complexity of the rendered model. Value is defined in MACC operations (Multiply-ACCumulated operations) (see [2])
rom (ro) indicates the requested size (in bytes) for the constant RO parameters (mainly bias and weights). Last value indicates the difference of parameter size as a percentage of the original parameter size. Size is 4-bytes aligned.
ram (rw) first value indicates the requested size (in bytes) for the working RW buffer. It is mainly used as internal heap for the activations (outputs of a given layer..). Second value indicates the expected size for the input and output tensors.

4 validate command

The validate command allows to import, to render and to validate a generated C-model. The full description of the underlying process can be found in [2], section “Validation engine”. Reported metrics to evaluate the generated C-model are fully described in [5]. When calculated the ‘L2r’ error by layer is appended in the complexity table.

4.1 Specific arguments.

Parameter Description
--mode indicates the mode of validation. x86 (default value) performs a validation on desktop. stm32 is used to perform a validation on target. - Optional
-vi/valinput indicates the custom test data set which must be used. If not defined an internal self-generated random data set is used (see “Input validation files” section in [5]) - Optional
-vo/valoutput indicates the expected custom output values. If the data are already provided in a simple file (*.npz) through the -vi option this argument is not considered - Optional
--batches indicates how many random data sample is generated (default: 10) - Optional
-d/--desc describes the COM port which is used to communicate with the STM32 board (see “desc argument” section) - Optional
--validate.batch_mode when an input custom data set is used, this argument is used to limit the number of samples. Two modes are possible: first to indicate that only the first batches samples are used. random indicates that batches samples are randomly selected with a fixed seed - Optional

4.1.1 desc argument

The --desc/-d argument should be used to indicate how to configure the COM driver with a STM32 board. The STM32 board should be flashed with the aiValidation application (see [2], section “AI validation application”).

By default, an auto detection mechanism is applied to discover a connected board at @115200 (default value: default:115200)

  • set the baud-rate to @961600

      $ stm32ai validate -m <model_file_path> --mode stm32 -d 921600

  • set the COM port to COM16 (Windows case ) or /dev/ttyACM0 (Linux case)

      $ stm32ai validate -m <model_file_path> --mode stm32 -d COM16
  $ stm32ai validate -m <model_file_path> --mode stm32 -d /dev/ttyACM0

  • set the COM port to COM16 and the baud-rate @921600

      $ stm32ai validate -m <model_file_path> --mode stm32 -d COM16:921600

4.2 Command examples

  • “Validation on desktop” of a floating-point model with self-generated random input data.

      $ stm32ai validate -m <model_file_path>

  • “Validation on target” of a floating-point model with self-generated random input data. The default aiValidation FW configuration: 115200 baud and network C-name is expected.

      $ stm32ai validate -m <model_file_path> --mode stm32

  • “Validation on desktop” of a floating-point model with self-generated random input data and compression factor.

      $ stm32ai validate -m <model_file_path> -c 4

  • “Validation on desktop” of a floating-point model with custom data set.

      $ stm32ai validate -m <model_file_path> -vi test_data.csv

  • “Validation on desktop” of a quantized model with custom data set

      $ stm32ai validate -m <modified_model_file>.h5 -q <quant_file_desc>.json -vi test_data.npz

6 Quantization and quantize command

X-CUBE-AI code generator can be used to generate and to deploy a pre-quantized Keras model. A reshaped model file (h5*) and a proprietary tensor format configuration file (json) are required. Note that the analyze, validate and generate commands can be used with the -q/--quantize argument to indicate the tensor format configuration file.

    $ stm32ai analyze -m <expanded_model_file>.h5 -q <quant_file_desc>.json

Code generator quantizes weights/bias and associated activations from floating point to 8-bits of precision. They are mapped on the optimized and specialized C-implementation for the supported kernels (see [3]). Otherwise the floating-point version of the operator is used and float-to-8-bits and 8-bits-to-float convert operators are automatically inserted. The objective of this technique is to reduce model size while also improving CPU and hardware accelerator latency (including power consumption aspects) with little degradation in model accuracy.

If the accuracy drop is too high, quantization aware training technique can be considered but not covered by the tools.

To generate the reshaped Keras model file and associated tensor format configuration file from an already-trained floating-point Keras model, the stm32ai application integrates a complete post-training quantization process (quantize command). In this case, the -q/--quantize argument is used to pass a quantization configuration file.

The proprietary tensor format configuration file gives the expected internal format for the tensors in the network, that is, the output and the weight/bias tensors. All the tensors for a layer must be in quantized format for the layer itself to be converted; otherwise, the formats of the weight tensors will be ignored. The only exceptions to this rule is when the format can be inferred from nearby layers. The formats for reshape layers (Flatten, Reshape, Permute) are inferred from the input or output layers. Reshaped model (see “Post-training quantization workflow” section) is requested for better quantization and implementation.

Note — By default, if a Softmax layer is part of the network, it is automatically keep in float.

6.1 Command examples

  • Perform the post-training quantization process on an already-trained floating-point Keras model.

    $ stm32ai quantize -q <conf_quant>.json

  • Validate a quantized Keras model.

    $ stm32ai validate -m <expanded_model_file>.h5 -q <quant_file_desc>.json -vi test_data.npz

  • Generate the specialized files for a quantized Keras model.

    $ stm32ai generate -m <expanded_model_file>.h5 -q <quant_file_desc>.json

6.2 Quantization configuration file

The requested quantization configuration file is a JSON text file with the following keys:

Key Description
“model_name” indicates the suffix used for the generated files.
- <model_name>.h5 - reshaped model file
- <model_name>_Q.json - tensor format configuration file
- <model_name>.npz - reference file
“path_to_floatingpoint_h5” indicates the path to the original model.
“algorithm” indicates the used algorithm. Possible values: “User”, “Greedy” or “Minmax” (see “Quantizer” section)
“quant_test_set_dir” indicates the quantization test-set directory (see “Test-set considerations” section).
“evaluation_test_set_dir” indicates the evaluation test-set directory (see “Test-set considerations” section).
“batch_size” indicates number of inputs vectors processed for one evaluation. The user needs to choose carefully this parameter in function of its system memory. Recommendation is to start with a small value for example 32.
“quant_test_ratio” indicates the ratio [0..1] of the vectors in “quant_test_set_dir” which are used for the quantization. They are randomly selected.
“output_directory” indicates the root directory to store the results. Generated files are stored in the following directory: <output_directory>/<model_name>_<algorithm>_<date>_<time>/.
“modules_directory” indicates the directory containing the user test-set generation and optional user quantizer Python files.
“filename_test_set_generation” name of the file (with or without py extension), where the user writes a potential pre-processing of the data, and in any case, load the test-sets into generators. This file is mandatory.
“filename_quantizer_algos” name of the file (with or without py extension), where the user can write his own quantizer. This is only needed if “User” algorithm is requested.

Note – For the “path_to_floatingpoint_h5”, “output_directory”, “modules_directory”, “quant_test_set_dir” and “evaluation_test_set_dir” keys if the values are not prefixed by "./" or "/", path is relative to the path of the JSON file path. Otherwise, the path is absolute or relative to the current executing path where stm32ai application is launched.

Note – All fields are mandatories. Only the “filename_quantizer_algos” can be omitted if “User” algo is not selected.

6.4 Test-set considerations

The quantization process gives the possibility to load batches of test vectors. The quantization requires the knowledge of the range of the values of all the tensors. The weights are constant and thus their range is easy to estimate, whereas the activation’s range depends on the input data. An accurate estimate of the activation range requires a test set representative of real data:

  • it means a balanced test set over all the labels to be classified
  • test set should be large enough to reflect how variable can be the network inputs and thus to be as representative as possible

We call it quantization test-set. If this test-set is too small, not representative enough or even unbalanced, quantization will be performed as usual but there is a risk that the quantized model performance be not satisfying. The other test-set, called evaluation test-set, is used to assess performance of floating point model and quantized model and potentially compare them. The evaluation test-set and the quantization test-set should be as independent as possible to avoid over-estimation of the quantized model performance.

The input vectors is loaded with the create_test_generator() function which is dynamically imported by the core from the user “filename_test_set_generation” module. Preprocessing or adaptation of the loaded data should be done in this function (see “Quantize a XXX model” sections).

def create_test_generator(quant_test_set_dir, evaluation_test_set_dir,
                          quant_test_ratio, batch_size)

Note — If the neural network is not used for a classification task, or if labels are not available for the quantizer then in the create_test_generator() function set quant_test_set_labeled to None and eval_test_set_labeled to None.

6.5 “Minmax”, “Greedy” and “User” quantizers

Two quantizers (also called algorithms) are provided to generate the tensor format configuration file: “Minmax” and “Greedy”. The user (“User” algorithm) has also the possibility to provide its own quantizer with different compromises between quantized data precision and saturation, by implementing the estimate method in the quantizer class and passing it to the functions for weight and activation quantization (see “filename_quantizer_algos” user module).

  • “Minmax” invokes the simple and quick quantization process based on min and max of all the tensors for weights and activations. The quantization test-set is used to estimate the activation ranges. The weights are constant and thus their ranges are directly estimated.

  • “Greedy” invokes an iterative process which tries to improve the picture for activations only, by searching on a per layer basis among a set of possible quantization formats which is the best in terms of accuracy. Note that only the accuracy is considered and not the loss function.
    • consequently, if there is no label in the quantization test-set then “Greedy” will not be executed.
    • be careful, the execution of this algorithm takes much more time than the simple “Minmax” but can also lead to better performance.

Whatever the algorithm used, the quantizer script issues the quantized model expected accuracy on the evaluation test-set (if labels are present). This is an interesting indication, that may be used to compare different quantizers but the recommendation is to verify that the quantized model generalize well-enough according to your requirements. This has to be done with real field inputs.
